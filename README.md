<details>
<summary><b>🔭 SearchTube — Production-Grade YouTube Search & Discovery Engine</b></summary>

<br/>

| Category | Details |
|----------|---------|
| **Architecture** | Multi-stage data extraction pipeline with layered search indexing, result normalization engine, and modular scraping infrastructure |
| **Technology Stack** | Python, BeautifulSoup4, Selenium, Requests, HTML Parsing, Data Processing |
| **Scalability** | Designed for large-scale metadata collection with parallel request handling and rate-limiting resilience |
| **Performance** | Optimized HTML parsing with targeted selector strategies, minimizing DOM traversal overhead |
| **Security** | User-agent rotation, request throttling, and session management to ensure reliable and responsible data collection |
| **Engineering Challenges** | Dynamic content rendering, anti-scraping mechanisms, result deduplication, and structured metadata normalization across varying page schemas |
| **Impact** | Delivers structured, ranked YouTube search results from raw web data — removing dependency on official API quota limits |
| **Repository** | [View Repository →](https://github.com/mrvanshgoel/Searchtube) |

---

**Problem Statement**

YouTube's official Data API imposes strict quota limitations that block high-volume data collection workflows. Engineers and researchers needing large-scale YouTube metadata — for analytics, content discovery, or recommendation research — are blocked by these constraints. SearchTube solves this by engineering a direct, scalable web extraction pipeline that operates independently of API quotas.

**System Design Overview**

```
[User Query]
     │
     ▼
[Search Request Engine] ──► [Session & Header Manager]
     │
     ▼
[HTML Fetcher (Requests + Selenium fallback for JS-rendered pages)]
     │
     ▼
[BeautifulSoup Parser] ──► [Element Selector Engine]
     │
     ▼
[Metadata Extractor: Title / Channel / Views / Duration / URL]
     │
     ▼
[Data Normalizer & Deduplicator]
     │
     ▼
[Result Ranker]
     │
     ▼
[Structured Output: JSON / CSV]
```

**Engineering Decisions**

- **Hybrid fetching strategy:** Static pages fetched with Requests for speed; JavaScript-rendered pages handled by Selenium as fallback — balancing performance with reliability.
- **BeautifulSoup for parsing:** Chosen over regex-based extraction for maintainability and robustness against minor HTML structure changes.
- **Modular pipeline design:** Each stage (fetch → parse → extract → normalize → rank) is independently testable and replaceable.

**Technical Challenges**

- YouTube's dynamic page rendering required a dual-strategy approach: Requests for fast static fetches and Selenium for JS-rendered content.
- Anti-scraping measures required engineering user-agent rotation and request throttling to maintain reliable collection at scale.
- Metadata normalization across varying YouTube page schemas required building a schema-agnostic extraction layer.

**Future Improvements**

- Distributed scraping with asynchronous queues (Celery + Redis)
- Elasticsearch-based result indexing for sub-millisecond search
- Real-time trend tracking with scheduled pipeline execution
- REST API layer for programmatic access to search results

<summary><b>🛡️ YT Mod Logger — Advanced YouTube Moderation Logging & Monitoring Platform</b></summary>

<br/>

| Category | Details |
|----------|---------|
| **Architecture** | Event-driven browser automation architecture with a real-time logging layer, automated workflow orchestration, and persistent event store |
| **Technology Stack** | Python, Selenium WebDriver, Browser Automation, Web Drivers, Logging Systems |
| **Scalability** | Designed for continuous, unattended operation with automated session recovery and state persistence |
| **Performance** | Optimized browser interaction sequences using explicit waits and element state validation to minimize flake |
| **Security** | Secure credential handling, session isolation, and browser fingerprint management |
| **Engineering Challenges** | Real-time DOM change detection, session persistence, event deduplication, and structured logging across dynamic browser states |
| **Impact** | Provides a reliable, automated audit trail of YouTube moderation events — critical for content creators and channel managers at scale |
| **Repository** | [View Repository →](https://github.com/mrvanshgoel/YT-ModLogger) |

---

**Problem Statement**

YouTube provides no native export or structured logging of moderation events. Content creators and channel administrators operating at scale have no reliable mechanism to audit what was moderated, when, or why. YT Mod Logger engineers this infrastructure directly — automating browser interaction to capture, structure, and persist a complete moderation event log.

**System Design Overview**

```
[Browser Session Initializer]
     │
     ▼
[Selenium WebDriver Controller]
     │
     ▼
[YouTube Session Manager] ──► [Auth & Cookie Handler]
     │
     ▼
[Page State Monitor] ──► [DOM Change Detector]
     │
     ▼
[Event Extractor: Moderation Action / Timestamp / Target / Status]
     │
     ▼
[Event Deduplicator]
     │
     ▼
[Structured Logger] ──► [Persistent Event Store (JSON / Log Files)]
     │
     ▼
[Audit Trail Output]
```

**Engineering Decisions**

- **Selenium WebDriver:** Chosen for full browser simulation capability — required for YouTube's authenticated, JavaScript-heavy moderation interface.
- **Event-driven architecture:** The logger monitors DOM state changes rather than polling at fixed intervals — reducing overhead and improving event capture accuracy.
- **Structured logging:** Events are serialized to structured formats (JSON, log files) for queryability and downstream processing.

**Technical Challenges**

- YouTube's moderation UI is heavily dynamic — reliable event capture required engineering robust DOM change detection with explicit wait strategies.
- Session persistence across long monitoring windows required automated session recovery logic to handle timeouts and re-authentication.
- Event deduplication was non-trivial due to YouTube's dynamic page re-rendering that could surface the same event multiple times.

**Future Improvements**

- Real-time alerting via webhook notifications (Slack, Discord, Email)
- Database backend (PostgreSQL) for queryable event history
- Dashboard UI for visual moderation audit trail
- Multi-channel concurrent monitoring support

</details>

| **Engineering Challenges** | API quota management, paginated response traversal, structured metadata normalization, and automated content classification |
| **Impact** | Automates high-volume YouTube content processing workflows — removing manual bottlenecks for content teams and data pipelines |
---

**Problem Statement**

Content teams and data engineers working with YouTube data at scale face a fundamental bottleneck: YouTube's API responses require significant processing to extract structured, usable metadata from raw JSON payloads. Manual processing is not viable at scale. ClipBot engineers an automated content processing pipeline that ingests, normalizes, and structures YouTube API data for downstream use.

**System Design Overview**

```
[YouTube Data API v3]
     │
     ▼
[API Client Layer] ──► [OAuth2 / Key Auth Manager]
     │
     ▼
[Paginated Response Handler] ──► [nextPageToken Iterator]
     │
     ▼
[JSON Response Deserializer]
     │
     ▼
[Metadata Extractor: Title / ID / Description / Stats / Tags / Timestamps]
     │
     ▼
[Data Normalizer & Validator]
     │
     ▼
[Automated Content Processor]
     │
     ▼
[Structured Output: JSON / CSV / Database]
```

**Engineering Decisions**

- **YouTube Data API v3:** Official API chosen for ClipBot to ensure reliable, structured data access — complementing SearchTube's web extraction approach for different use cases.
- **Quota-aware batching:** API calls structured as batched requests to maximize data throughput within quota constraints.
- **Paginated traversal:** Automated `nextPageToken` iteration handles arbitrarily large result sets without manual intervention.

**Technical Challenges**

- YouTube's API quota system required engineering a quota-aware request scheduler that balances throughput against daily limits.
- Paginated API responses required a robust traversal engine to handle large datasets without data loss or duplication.
- JSON payloads vary significantly across video types and API endpoints — requiring a schema-tolerant normalization layer.

**Future Improvements**

- Quota monitoring dashboard with real-time usage tracking
- Async processing with concurrent API requests

- ML-powered content classification on extracted metadata

<summary><b>📱 NoteSphere — Cross-Platform Notes Management System</b></summary>

<br/>

| Category | Details |
|----------|---------|
| **Architecture** | Full-stack architecture with a Node.js/Express.js REST API backend, Java Android client, SQLite persistence layer, and data synchronization engine |
| **Technology Stack** | Node.js, Express.js, Java, Android SDK, SQLite, REST APIs |
| **Scalability** | RESTful backend designed for horizontal scaling; Android client built for offline-first operation with sync |
| **Performance** | Lightweight SQLite persistence on mobile; optimized REST endpoints with minimal payload sizes |
| **Security** | User authentication, JWT-based session management, and input validation on all API endpoints |
| **Engineering Challenges** | Cross-platform data synchronization, offline-first mobile design, conflict resolution, and REST API design for mobile consumption |
| **Impact** | Delivers a complete, production-grade notes management system across platforms — demonstrating full-stack and mobile engineering capability |
| **Repository** | [View Repository →](https://github.com/mrvanshgoel/NoteSphere) |

---

**Problem Statement**

Consumer note-taking apps are black boxes — no API access, no data portability, no developer control. NoteSphere is engineered from the ground up as a full-stack system: a REST API backend that owns the business logic and data layer, and a native Android client that consumes it — giving engineers full ownership of the entire stack.

**System Design Overview**

```
[Android Client (Java)]
     │
     ▼
[REST API Client Layer] ──► [JWT Auth Header Manager]
     │
     ▼
[Express.js REST API (Node.js)]
     │
     ├──► [Auth Router: /register, /login]
     ├──► [Notes Router: /notes CRUD]
     └──► [Sync Router: /sync]
          │
          ▼
     [SQLite Database Layer]
          │
          ▼
     [Data Models: User / Note / Metadata]
```

**Engineering Decisions**

- **Node.js + Express.js backend:** Chosen for rapid API development, JSON-native data handling, and lightweight runtime suitable for cloud deployment.
- **Java Android client:** Native Android development for full access to Android SDK capabilities and optimal mobile performance.
- **SQLite:** Lightweight, serverless database chosen for simplicity and portability in the initial architecture — designed for migration to PostgreSQL at scale.
- **REST API design:** Standard CRUD operations with JWT authentication provide a clean, stateless API contract between client and server.

**Technical Challenges**

- Designing an offline-first Android client that synchronizes reliably with the backend required careful conflict resolution logic for concurrent edits.
- Cross-platform data consistency between the Android SQLite local store and the backend database required engineering a robust sync protocol.
- JWT session management required handling token refresh, expiry, and secure storage on the Android client.

**Future Improvements**

- PostgreSQL backend for production-scale deployment
- Real-time sync with WebSocket connections
- Cloud deployment on AWS EC2 with HTTPS via Nginx reverse proxy
- Rich text editor and attachment support in the Android client

</details>

## ◈ Engineering Philosophy

<div align="center">

> *"A system's quality is determined not by what it does when everything goes right, but by how it behaves when something goes wrong."*

</div>

Engineering is a discipline of trade-offs. Every architectural decision — a choice of database, a communication protocol, a data model — carries consequences that compound over time. I approach software with the belief that good engineering is **invisible**: users should never see the infrastructure, only the outcome.

My philosophy is built on three principles:

**Correctness before optimization.** A fast system that produces wrong results is not a system — it is a liability. I validate assumptions, write defensive code, and test edge cases before I measure performance.

**Simplicity as architecture.** Complexity is the primary source of bugs, outages, and maintenance debt. I design for the simplest architecture that correctly solves the problem at its required scale — and no simpler.

**Ownership through the full lifecycle.** Real engineering is not writing code — it is defining the problem, designing the solution, building it, deploying it, monitoring it, and iterating on it. I build systems I would be comfortable operating at 3am.
